## V1（周二 20:30）

### 题目
`30秒讲清：为什么别一上来就做全自治 Agent？`

### 封面文案
- 主标题：`先简单，再自治`
- 副标题：`90%团队第一步就做错了`

### 30 秒口播稿
1. 先说结论：别一上来就做全自治 Agent。  
2. 原因很简单：任务可预测时，Workflow 更稳、更便宜。  
3. 只有当问题不确定、步骤会变化，Agent 才能体现价值。  
4. 真正的顺序是：先跑通，再自治。  
5. 这不是保守，是工程最优策略。  
6. 想看“如何判断一个任务该用 Workflow 还是 Agent”，评论区打“边界”。

### 结构模板
- 0-3 秒：结论先行
- 3-18 秒：对比（Workflow vs Agent）
- 18-26 秒：一个现实场景
- 26-30 秒：CTA

### CTA 模板
- `评论“边界”，我发你任务判断清单。`

## V2（周四 20:30）

### 题目
`Agent Loop 不是口号：Gather/Act/Verify 到底怎么落地？`

### 封面文案
- 主标题：`三步闭环`
- 副标题：`不验证=高概率翻车`

### 45 秒口播稿
1. 很多 Agent 项目失败，不是模型不够聪明，而是没有闭环。  
2. 闭环就三步：Gather、Act、Verify。  
3. Gather 是收集关键上下文，不是把所有信息都塞进去。  
4. Act 是执行动作：调用工具、脚本、外部接口。  
5. Verify 才是关键：用规则、测试和对照去确认结果。  
6. 没有 Verify，模型说“完成”并不等于真的完成。  
7. 你可以今天就做一件事：给任务加一个可验证检查点。  
8. 评论“闭环”，我发你最小闭环模板。

### 结构模板
- 0-4 秒：失败痛点
- 4-28 秒：三步解释
- 28-40 秒：立即可执行动作
- 40-45 秒：CTA

## V3（周六 20:30）

### 题目
`本周避坑：3个“看起来很智能、实际很脆弱”的 Agent 做法`

### 封面文案
- 主标题：`3个高频坑`
- 副标题：`聪明感≠可交付`

### 45 秒口播稿
1. 第一个坑：把 Agent 当长 prompt，不做工具化。  
2. 第二个坑：只看演示成功，不做失败路径。  
3. 第三个坑：不看成本和重试率，只看结果“像不像”。  
4. 对应修正也很简单：  
5. 坑1加工具契约，坑2加回归检查，坑3加成本指标。  
6. 真正可落地的 Agent，一定是可验证、可回滚、可复盘。  
7. 收藏这条，做你下周的项目自检表。

### 结构模板
- 0-3 秒：痛点钩子
- 3-30 秒：误区 1/2/3
- 30-40 秒：修正动作
- 40-45 秒：CTA
